# 安全处理手册

这份手册面向本地部署和小团队协作环境，重点处理最常见、最真实的三类安全问题：`token` 泄露、账号异常、恶意脚本/查询。

## 1. Token 泄露

### 识别信号

- 用户反馈账号被异常登录或项目被异常修改。
- 日志里出现未知客户端的频繁 `auth_login`、`project_commit`、`dataset_upload`。
- 同一 `user_id` 在短时间内从多个不符合预期的来源持续操作。

### 处理步骤

1. 先暂停当前操作窗口，确认是否仍有未授权请求持续发生。
2. 让用户立即重新登录，并在后端执行 token 失效处理。
3. 检查最近审计日志，确认异常操作范围：
   - `request_id`
   - `user_id`
   - `project_id`
   - `action`
4. 如果影响到项目状态或数据集：
   - 导出当前项目状态与数据目录备份
   - 核对最近一次正常版本
   - 必要时回滚到上一个可信版本
5. 让用户更换密码，并检查是否存在硬编码 token、截图泄露、共享日志等问题。

### 建议补救

- 清理本地浏览器存储中的旧 token。
- 不要把 token 写进脚本、截图、聊天记录或提交到仓库。
- 对外分享日志前先脱敏。

## 2. 账号异常

### 典型场景

- 登录失败次数突然增多。
- 用户被加入陌生项目或组织。
- 成员角色被异常提升。

### 处理步骤

1. 查看最近的登录失败与成员变更审计日志。
2. 确认是否为用户自己在多端测试或脚本误操作。
3. 如果怀疑被暴力尝试：
   - 暂时停用异常账号的操作
   - 清理现有登录态
   - 让用户重置密码
4. 如果已经发生越权成员变更：
   - 立即移除异常成员
   - 复核项目成员角色
   - 检查最近的项目提交和数据集操作

### 需要保留的信息

- 异常发生时间
- 涉及的用户邮箱
- 相关 `request_id`
- 受影响的 `project_id` / `organization_id`

## 3. 恶意脚本、上传或查询

### 风险来源

- 伪装成数据文件的可执行脚本
- Python transform 中试图调用危险内建
- SQL 查询中夹带写操作或多语句

### 处理步骤

1. 如果上传被拦截：
   - 保留原始文件名和 MIME 信息
   - 不要把文件二次分发给其他同事
2. 如果 transform 或 SQL 被拦截：
   - 记录报错信息和 `request_id`
   - 确认是否是误报还是确实含危险操作
3. 如果怀疑已有恶意内容落地：
   - 先备份当前 `collab db + sessions + project_assets`
   - 检查最近新增的数据集与项目提交
   - 必要时删除异常数据并从备份恢复

### 默认原则

- 不为“先跑通再说”临时打开危险执行开关。
- 如果必须临时放宽限制，只在本机短时开启，并记录原因与结束时间。
- 放宽后必须恢复默认安全配置，并补一轮回归测试。

## 4. 最小排障清单

每次处理安全问题，至少确认下面几项：

1. 当前认证模式是否符合预期。
2. 最近 15 分钟内有没有异常登录失败。
3. 最近一次项目提交和数据集上传是谁发起的。
4. SQL / transform 是否触发了安全拦截。
5. 是否已经做了备份，能否回退。
